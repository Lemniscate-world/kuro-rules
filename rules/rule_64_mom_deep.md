# RULE 64: Negative Mom Test - Deep Verification Protocol

## Trigger: Mom Test returns negative signals.
## DO NOT immediately declare the project dead.

## Step 1: Classify the Rejection Type
| Type | Signal | Meaning |
|------|--------|---------|
| A - Wrong person | "I do not have this problem" | Wrong target persona, not wrong problem |
| B - Wrong framing | "I do not understand what you mean" | Problem exists but framed incorrectly |
| C - Wrong timing | "This was a problem 2 years ago" | Problem solved by market, pivot needed |
| D - Real rejection | "I have this but would never pay" | Genuine NO-GO signal |
| E - Polite rejection | "Interesting idea!" with no story | Not a real rejection, re-interview |

## Step 2: Deep Verification (for Type A, B, C only)
1. Reframe the persona: Who DOES have this problem? Interview 3 of them.
2. Reframe the problem: What is the root cause?
3. Desk Research: Search Reddit/HN with different keywords.
4. Competitor signal: Do competitors exist? If yes, problem is real.

## Step 3: Evidence Threshold
Continue if you find at least 2 of:
- 3+ Reddit threads with 50+ upvotes about this pain
- 1+ competitor with active users
- 2+ people who built a workaround
- 1+ job posting mentioning this as required skill

## Step 4: Final Decision Matrix
Type D rejection (2+ instances) -> NO-GO, document in KNOWLEDGE_BASE/
Type A + reframed persona finds pain -> PIVOT persona, not idea
Type B + reframed problem finds pain -> PIVOT framing, not idea
Type C -> PIVOT to adjacent current problem
All evidence missing after deep verification -> NO-GO

## Enforcement
IF Mom Test returns negative:
  DO NOT: Immediately declare project dead
  DO: Classify rejection type first
  DO: Run deep verification for Type A/B/C
  DO: Document findings either way
