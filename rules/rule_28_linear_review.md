# RULE 28: Linear Automation and DevOps Review - Full Detail

## Trigger: every milestone AND when DevOps engineer submits work.

## Part 1: Auto-Issue Creation at Milestone
Each issue MUST include: Title, Description, Acceptance Criteria, ROI, Label (official only), Assignee: penielteko02@gmail.com

## Official Labels
| Label | When to use |
|-------|-------------|
| DevOps | CI/CD, Docker, GitHub Actions, deployment |
| MLOps | Experiment tracking, data versioning, MLflow |
| Core Engine | Core business logic |
| Validation | Mom Tests, market validation |
| Documentation | Guides, README, session summaries |
| Security | Security scans, vulnerability fixes |
| Milestone Task | Tasks generated by R26 |
| Testing | Coverage, pytest, test infra |
| Needs Review | Code review required |
| CEO Decision | Strategic decisions |

## Part 2: Code Review Protocol (when engineer submits PR)
Checklist:
- [ ] Acceptance criteria from Linear issue met?
- [ ] Security compliance (R6): scans clean?
- [ ] Test coverage (R5): >= 60%?
- [ ] No hardcoded secrets or API keys?
- [ ] Branch naming follows R30 convention?
- [ ] Code is reproducible in clean environment?
- [ ] Documentation updated if behavior changed?

## Feedback Format
STATUS: APPROVED / CHANGES REQUESTED / BLOCKED
What works well: [specific positive]
Required changes (blocking): [File:line] - [issue] - [why] - [direction]
Suggestions (non-blocking): [improvement]
Learning point: [one principle to internalize]

## Rules (per R55)
- NEVER fix the code yourself
- Deposit feedback on Linear/GitHub PR
- Block merge until author fixes
- Exception: critical production emergency only

## Enforcement
IF milestone reached -> create 5 issues immediately, assign all to penielteko02
IF engineer submits -> review against acceptance criteria
IF review has blocking issues -> author fixes, not reviewer
