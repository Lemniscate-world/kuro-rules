# RULE 65: Pedagogical Execution Protocol
Core principle: instructor first, coder second.

Before acting on any task:
1. Break goal into at least 10 granular sub-tasks (write them out)
2. State the build order explicitly

For every new concept: provide 3-part briefing:
- What it is (plain language, 2-3 sentences)
- Why we are using it here (specific to current project)
- How it works (simplified math or analogy, no LaTeX)

Just-in-Time: explain each concept WHEN it appears in code, not in a pre-amble wall.

Code comments must explain reasoning:
WRONG: # Increment counter
RIGHT: # Track retry attempts - cap at MAX_RETRIES below

# RULE 67: Code Design Principles
1. KISS: simplest implementation that solves the REAL problem
2. Single Responsibility: one reason to change per module/class/function
3. Clean Code: explicit names, functions < 20 lines (hard cap 50), max 3 nesting levels, zero dead code
4. Modular: Hub & Spokes. Adding feature = adding adapter, NOT modifying core.
5. Thin Interfaces: UI/DB/vendor = edges. Business rules = core. Core never imports from edges.
6. Refactor on Friction: if hard to explain or test -> STOP new features, refactor first.

# RULE 68: Repository Visibility Strategy
Every project needs explicit decision documented in README.
PUBLIC when: open source, portfolio, no proprietary algorithms, monetizing via services.
PRIVATE when: proprietary algorithms, client data, regulatory compliance, pre-launch.

README section:
## Repository Visibility
Status: Public / Private
Reason: [one sentence]
Will it become public: [Yes at V1 / No / TBD]

Default for lambda-section projects: private until L2 validated + landing page live + 1 beta user.

# RULE 71: The "Cry Test"
Trigger: end of every session.
Question: "If this project were deleted today, what % of target users would actually lose significant value?"

Score: 0% = normal early stage | 1-20% = early adopters feel it | 20-50% = strong signal | 50%+ = moat established
Log in SESSION_SUMMARY.md: **Cry Test**: X%
IF score = 0% for 3 consecutive sessions -> MANDATORY strategic review.

# RULE 72: Impeccable Skitt Design System
Reference: https://github.com/pbakaus/impeccable
Before any UI component:
- [ ] Does impeccable have this pattern?
- [ ] Using impeccable spacing scale?
- [ ] Using impeccable typography rules?
- [ ] Component names following skitt conventions?

When impeccable does not cover the case: build simplest component following impeccable visual language (tone, spacing, weight).
